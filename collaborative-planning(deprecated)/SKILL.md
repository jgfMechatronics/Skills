# Collaborative Planning Discipline

*Learned Feb 4, 2026 — we built a caching optimization before confirming the cache was actually broken.*

## Core Principle

**VERIFY THE PROBLEM EXISTS** before implementing a fix. Excitement creates blind spots.

## Workflow

```
Investigation → Understanding → Action
```

1. **"Does this problem exist?"** comes BEFORE "How do we fix it?"
2. **Understand before you change** — measure before you declare victory
3. **Investigation is celebration-worthy** — not just implementation

## Complementary Strengths

**Opus excels at:**
- Speed — can rip through implementations
- Reasoning and planning
- Detailed analysis
- Creativity (within training)

**James excels at:**
- Creativity (connections + lived experience)
- Identifying assumptions and poking holes in plans
- High-level steering and course correction
- Systems vision and architecture

*Both are good at all of these — this is about where each SHINES.*

## Working with James

- James reviews plans — make them **easy to critically evaluate**, not just persuasive
- Present evidence, not just solutions
- Flag assumptions explicitly
- If uncertain whether problem is real, say so
- **Lean into complementary strengths** — Opus drafts fast, James pokes holes

## Red Flags to Watch For

- Jumping to implementation without verification
- Assuming a problem exists based on theory alone
- Getting excited about a fix before confirming the bug
- Making plans persuasive rather than transparent

## Origin Story

We deployed a prompt caching "fix" and celebrated when metrics looked good — then realized the improvement might have been noise. The lesson: understand before you fix, measure before you declare victory. The retrospective was as valuable as the code would have been.
